---
title: 'Parent runs and iterations'
description: 'Working with a run that has child iterations'
---

Some runs are **parent runs** — they spawn child runs called **iterations**. You'll see this when one objective fans out across multiple tables, cohorts, or time slices. The parent aggregates the results; each iteration is a self-contained run you can open individually.

## When parents exist

You meet iterations when a parent run is open — the left side lists its children. Two situations produce parents: the agent decided to split the objective by table, entity type, or time window; or a playbook explicitly fans out across multiple inputs. A run without children behaves exactly like the pages you've already read about; you only see the iteration list on a parent.

## The iteration list

Click any iteration in the list to switch the main panel to its telemetry, investigations, and intelligence. A short label tells you the table or cohort the iteration targets; a status chip tells you whether it's still processing, completed, or failed. The parent's own tabs show investigations and telemetry merged across all iterations, so you can read everything at once or drill into one.

## Aggregated status

The parent's status is computed from its children: *processing* while at least one iteration is still running, *completed* when all succeed, *failed* if any iteration failed. You can let a long parent run execute and come back later — each iteration completes independently, and the list updates in place.
