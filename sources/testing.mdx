title: 'Testing a connection'
description: 'Use Test source to confirm Conway can reach your warehouse'
---

After you connect a warehouse, run a **test** before trusting it with a real run. The test confirms Conway can reach the warehouse, authenticate, and list tables in the configured scope.

## Running a test

On the source card on the [Sources](/sources/overview) page, click **Test source**. Conway opens a connection to the host, authenticates with the saved credentials, lists the configured database/schema/tables, and runs a trivial query against one of them. The card overlays progress while the probe runs, turning green on success or red on failure.

On success, the source is ready to use. You don't need to re-test until something changes — credentials rotate, the schema moves, the user's role changes.

## Failures

A red overlay carries a short error message; click it to expand the full message from the warehouse. Common causes:

- **Authentication failed** — wrong password, expired OAuth token, key-pair mismatch. Re-open the source and re-enter the credentials.
- **Missing privileges** — the user/role can authenticate but can't see the database, schema, or tables. Use **Edit scope** on the source card, or have the warehouse admin widen the role.
- **Network unreachable** — the warehouse isn't reachable from Conway's network. This is almost always an admin problem; contact them.
- **Latency too high** — the probe timed out. Often transient; retry first. If it persists, escalate to the admin.

To fix a failing test:

1. Read the expanded error — it almost always names the specific step that failed.
2. If credential-related, re-open the source and re-enter them.
3. If scope-related, use **Edit scope** to adjust the default database/schema.
4. If network-related, contact your admin.

<Note>
In **Pilot deployments**, a failing test should go straight to your Conway contact rather than a self-service fix. See [Pilot limitations](/pilot/limitations).
</Note>
